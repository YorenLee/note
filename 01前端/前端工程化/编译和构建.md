## 为什么需要编译？

早期浏览器只支持es5、简单css、没有模块系统，而我们现代写的`Jsx`、`Tsx`、`Es6`、`装饰器`浏览器不能识别，此时需要编译将我们写的代码转化为浏览器能认识的代码

## 编译工具

### Babel(JS)
####  Babel慢的原因
```
源码  
↓  
Parse（生成 AST）  
↓  
Transform（插件一轮一轮改 AST）  
↓  
Generate（生成代码）
```
1.  **Transform 不是一轮，而是很多轮** 比如有`JSX 插件`  `TS 插件` 等等 `每个插件都要全量遍历 AST`
2. 是Js实现
	-   内存分配慢：AST 是大量对象 JS对象分配慢，GC压力大
	- 单线程执行 AST 遍历
#### Babel优势
1. 精准语法降级
2. 按需polyfill (core-js)
3. 插件生态
	- 几乎可以改任何js
		  ```
				  visitor: {
					  Identifier(path) {
						// 可以随便改代码
						  }
					}
		  ```
### SWC (Rust)
#### 为什么快
1. 基于Rust 写的可以多线程 并且原生性能较快接近c++
2. 所有语法转换在 **一次 AST 遍历中完成**
3.  静态分析优先：SWC 不允许插件随意动态修改 AST 所以可以提前优化、减少对象创建
4. 内存高效： AST 节点使用 Rust struct，而不是 JS 对象、减少 GC 压力
5. 单次流水线 + 并行化：可以用 Rust 的线程池并行处理多个文件
	```
		SWC 可以一次处理：
		TS → JS
		JSX → JS
		ESNext → ES5
	```
	不需要多轮插件

## 打包
有的打包工具也支持编译
打包工具本质上就是把项目的源码、依赖、资源等整理成浏览器可以直接执行的文件。

| 工具      | 类型      | 核心特点                                |
| ------- | ------- | ----------------------------------- |
| Webpack | 模块打包器   | 功能最全，插件和 loader 生态丰富，兼容性强           |
| Rollup  | 模块打包器   | 输出干净的 ES Module，适合库和包               |
| Parcel  | 零配置打包器  | 开箱即用，自动处理依赖和资源                      |
| Vite    | 开发+构建工具 | 开发极速热更新 + 构建时调用 Rollup 或 SWC，现代前端首选 |
| esbuild | 高性能打包器  | 超快编译和打包，但插件生态有限                     |

### 打包核心职责
- **模块解析**：JS/TS/ESM/CJS
- **依赖分析**：构建依赖图
- **文件打包**：JS/CSS/图片/静态资源
- **代码优化**：Tree-shaking / 压缩 / Polyfill
- **开发体验**：HMR / Dev Server
