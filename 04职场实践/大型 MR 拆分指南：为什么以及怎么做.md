  

> 适用于不同经验水平的工程师。无论你是第一次提交 MR 的新人，还是主导架构重构的资深工程师，都能从中找到适合自己的实践方法。

## 第一部分：为什么 MR 不能太大

### 1.1 审查质量与改动规模的关系


Cisco 与 SmartBear 联合进行的代码审查研究（[Best Kept Secrets of Peer Code Review](https://smartbear.com/learn/code-review/best-practices-for-peer-code-review/)）给出了一个被业界广泛引用的结论：  

- 审查 **200-400 行** 代码时，缺陷发现率最高（约 70-90%）
    
- 超过 **400 行** 后，发现率急剧下降
    
- 超过 **1000 行** 时，审查者的大脑会进入「扫视模式」——看起来在逐行审查，实际上只是在滚动页面
    


Google 的工程实践文档（[Google Engineering Practices - Small CLs](https://google.github.io/eng-practices/review/developer/small-cls.html)）更是直接将「保持 CL 尽可能小」列为开发者的首要职责之一，其中 CL（Changelist）等同于我们说的 MR / PR。

这不是审查者不认真，而是人类认知的固有限制。当一个 MR 包含 3000 行改动时，审查者的心理状态大致是这样的：


```Plain
前 200 行：仔细看，提出有价值的反馈
200-800 行：开始疲劳，只关注明显问题
800 行以后：LGTM 🚢
```


**一个 3 万行的 MR，本质上是一个没有被审查的 MR。**

### 1.2 大 MR 的六个具体危害


#### 危害一：审查变成走过场


审查者打开一个 47 个文件变更的 MR，内心会产生畏惧感。为了不阻塞团队进度，他们倾向于快速浏览后点击 Approve。这不是审查者的问题，而是你把不合理的任务推给了别人。


Google 的研究数据显示（[Speed of Code Reviews, Google Eng Practices](https://google.github.io/eng-practices/review/reviewer/speed.html)），小型 CL 的审查响应中位数在数小时内，而大型 CL 的审查常常被推迟数天——因为没有人愿意在工作间隙打开一个需要 2 小时才能看完的 MR。


#### 危害二：Bug 的天然藏身之处


改动越多，Bug 越容易躲在审查者的注意力盲区里。一个 50 行的 MR 中，每一行都会被仔细审视；一个 5000 行的 MR 中，一个关键的边界条件错误（例如循环少了一次、数组下标差了一位）可能就藏在第 3847 行，没人会注意到。


**真实案例**：Node.js VFS PR（[#61478](https://github.com/nodejs/node/pull/61478)）约 2 万行改动，审查者 ThanhDodeurOdoo 在审查数周后才[发现一个设计层面的严重问题](https://github.com/nodejs/node/pull/61478#discussion_r1961428851)——`fs.open()` 在 VFS 路径下返回的是一个对象而不是数字类型的文件描述符，这会破坏所有假定 fd 是数字的代码。如果 `fs.open()` 的集成是一个独立的小 MR，这个问题很可能在第一轮审查就被发现。


#### 危害三：合并冲突雪崩


一个存活 3 周的分支，主干上可能已经有了上百次提交。你的每一行改动都可能与别人的改动产生冲突。分支存活时间越长，合并时遇到的冲突不只是数量增加，而且处理难度也成倍上升——因为你改了 A 文件，别人也改了 A 文件，而你们各自的改动又依赖了 B、C、D 文件中的不同版本。


持续集成的核心理念（参见 Martin Fowler 的 [Continuous Integration](https://martinfowler.com/articles/continuousIntegration.html)）就是「频繁地将代码集成到主干」。大 MR 与这一理念根本对立。


#### 危害四：回滚代价极高

  
线上出了问题，你需要回滚。如果问题来自一个 50 行的 MR，回滚后影响范围清晰可控。如果问题来自一个 3000 行的 MR，回滚意味着同时撤销了其中 2950 行正确的代码，而这些代码可能已经被后续的 MR 所依赖——导致连锁回滚。


#### 危害五：阻塞团队协作


你有一个 3 万行的 MR 在等审查。与此同时，团队中 3 个人需要用到你写的某个工具函数。他们只能选择：等你合入（阻塞数天）、复制你的代码（产生重复）、或者基于你的分支再开分支（产生嵌套依赖）。每一种都是坏选择。


#### 危害六：上下文丢失


你写这 3 万行代码用了 3 周，你对每一个决策的记忆都是鲜活的。但审查者看你的 MR 时，他需要在几个小时内重建你 3 周的心智模型。这几乎不可能做到。小 MR 让审查者只需理解一个小范围的上下文，审查反馈也更有针对性。

### 1.3 反直觉的事实：小 MR 的总时间更短

很多人不愿意拆分 MR 的理由是「拆成 10 个 MR 不是更慢吗？」

事实恰好相反。以下对比参考了 Google 的工程实践数据和 [LinearB 对数千个团队的分析](https://linearb.io/blog/2025-engineering-benchmarks-insights)：

|   |   |   |   |   |
|---|---|---|---|---|
|方式|审查者数量|每个 MR 审查时间|等待周期|总合入时间|
|1 个大 MR（3000 行）|需要 2-3 个资深审查者|2-4 小时（常被推迟）|3-7 天|1-2 周|
|10 个小 MR（300 行）|1 个审查者即可|15-30 分钟|当天或次日|3-5 天|

小 MR 可以利用审查者的碎片时间：等构建的 10 分钟、午饭前的 20 分钟、两个会议之间的 15 分钟。没有人愿意在碎片时间里打开一个 47 个文件的 MR。

## 第二部分：MR 的合理规模


### 2.1 经验法则


|   |   |   |
|---|---|---|
|指标|建议范围|备注|
|改动行数（不含测试）|**200-500 行**|超过 800 行需要有充分理由|
|改动文件数|**1-10 个**|超过 15 个文件很可能职责不单一|
|审查所需时间|**15-45 分钟**|超过 1 小时审查者容易走神|
|MR 描述长度|**一段话能说清做了什么**|如果描述需要写 2 页，说明 MR 太大|

Google 的建议（[Small CLs - Google Eng Practices](https://google.github.io/eng-practices/review/developer/small-cls.html)）是：**一个 CL 应该是一个最小的、独立的、完整的变更**。「独立」指它可以单独被审查和理解，「完整」指它不会让代码处于中间不可用状态。

### 2.2 例外情况


以下情况允许更大的 MR，但需要在 MR 描述中说明原因：


- **自动生成的代码**（如 API 客户端、数据库模型、图标文件）：标注哪些是生成的，审查者可以跳过
    
- **批量重命名或移动文件**：一个重命名操作可能涉及几十个文件，但逻辑变更为零
    
- **新增独立模块**：如果是全新的、不与现有代码耦合的模块，审查负担较低
    
- **纯测试补充**：为已有代码补充测试，运行时行为没有变化，风险较低

## 第三部分：拆分策略


### 策略一：按架构层次拆分（自底向上）


**适用场景**：新增子系统、基础组件开发、底层库替换。


核心思路是按依赖方向拆分——先合入被依赖的底层，再合入上层。就像建房子：先打地基，再砌墙，最后装屋顶。


```Plain
PR 1: 类型定义 + 接口声明
      → 零副作用，只定义"契约"

PR 2: 核心数据模型 / 工具函数
      → 实现核心逻辑，可独立编写单元测试

PR 3: 业务逻辑层
      → 依赖 PR 2 的接口

PR 4: 集成层（与现有系统对接）
      → 把新系统接入已有架构

PR 5: UI / 入口层
      → 用户可见的变更
```


**实际案例**：Node.js VFS 的作者在 Review Guide 中自己划分了 10 个子系统（数据模型 → 文件系统 → 注入层 → 文件描述符 → 模块加载器 → 流与监听 → SEA 集成 → Mock API 等），每一个完全可以是一个独立 PR。他后来不得不把 PR 转回 Draft 状态，花额外时间写 Review Guide 帮助审查者理解结构——如果一开始就拆分成 10 个 PR，这份 Guide 就是多余的，因为每个 PR 本身就是自解释的。


### 策略二：Feature Flag 保护下的增量合入


**适用场景**：大功能开发，需要长期迭代但又要频繁合入主干。


Feature Flag（功能开关）是一种在代码中通过配置控制某个功能是否对用户可见的技术。开关关闭时，新代码虽然已经合入主干，但不会对用户产生任何影响。


```TypeScript
// PR 1: 引入 feature flag + 类型定义
const ENABLE_NEW_EDITOR = process.env.NEXT_PUBLIC_FF_NEW_EDITOR === 'true';

// PR 2: 新编辑器的基础组件（flag 保护下，用户看不到）
export const NewEditor = () => {
  if (!ENABLE_NEW_EDITOR) return null;
  return <EditorCore />;
};

// PR 3-8: 逐步实现子功能，每个 PR 独立可审查
// PR 9: 开启 flag，用户可见
// PR 10: 清理 flag 相关代码
```


**实际案例**：

- **React Fiber 重写**（[React 16 发布博客](https://legacy.reactjs.org/blog/2017/09/26/react-v16.0.html)）：Facebook 用了约 2 年时间，在 feature flag 保护下通过数百个小 PR 完全重写了 React 的核心渲染引擎（协调器），期间 React 15 正常发布和维护，用户零感知
    
- **Next.js App Router**（[Next.js 13 发布博客](https://nextjs.org/blog/next-13)）：Vercel 通过 `appDir` 实验性 flag，让新路由系统与旧路由系统并存超过一年，期间持续向主干合入改动
    

### 策略三：绞杀者模式（渐进式替换）


**适用场景**：框架迁移、大规模重构，新旧系统需要共存一段时间。


这个名字来源于热带雨林中的绞杀榕（Strangler Fig）——它不砍倒旧树，而是缠绕在旧树上逐渐生长，最终完全替代旧树。Martin Fowler 将这个比喻[引入了软件工程领域](https://martinfowler.com/bliki/StranglerFigApplication.html)。


```Plain
阶段 1 [1 个 PR]: 引入中间适配层
      旧代码 → 适配层 → 旧实现（运行行为完全不变）

阶段 2 [N 个 PR]: 新模块用新实现，通过适配层接入
      旧代码 → 适配层 → 旧实现
      新代码 → 适配层 → 新实现

阶段 3 [N 个 PR]: 逐个迁移旧模块到新实现
      所有代码 → 适配层 → 新实现

阶段 4 [1 个 PR]: 移除适配层和旧代码
      所有代码 → 新实现（直接调用）
```


**实际案例**：

- **React Class 组件 → Hooks 迁移**：不是一次性重写所有组件，而是新组件用 Hooks 编写，旧组件按优先级逐个迁移，两种写法可以在同一个项目中长期共存
    
- **数据库迁移（双写模式）**：先同时向新旧两个数据库写入数据，然后逐步将读取流量切换到新数据库，确认无误后停止向旧数据库写入
    


### 策略四：先达成共识，再开始编码


**适用场景**：涉及架构决策、存在多种可能方案的改动。


```Plain
PR 0: RFC / 设计文档（纯文档，不含代码）
      → 收集团队反馈，就技术方案达成共识
      → 明确拆分计划和每个 PR 的范围

PR 1-N: 按共识方案逐步实现
```

RFC（Request for Comments）是工程团队中常用的设计提案流程——作者写出方案文档，团队成员书面评审后达成共识，然后再开始编码。Rust 语言的所有重大特性都通过 [RFC 流程](https://github.com/rust-lang/rfcs) 推进。

**反面案例**：VFS PR 中，Qard 提出了完全不同的架构方向——他认为应该用依赖注入（即把虚拟文件系统作为参数传递给需要它的函数）而不是全局挂载（即默默替换掉所有 `fs` 操作的行为）。arcanis 则用 Yarn PnP 6 年的生产验证经验支持全局挂载方案。这个根本性的设计讨论发生在 2 万行代码已经写完之后——如果事先走了 RFC 流程，可以避免大量可能被推翻的工作。


### 策略五：提取「无行为变更」的准备性 PR


**适用场景**：任何大改动都可以作为第一步。

在实现功能之前，先提交不改变任何运行行为的 PR：

```Plain
PR 1: 纯重构——提取函数、拆分文件、调整目录结构
      → 程序运行前后行为完全一致，容易审查
      → 降低后续 PR 的差异噪音

PR 2: 类型定义和接口声明
      → 只定义新系统的"契约"，审查者只需关注 API 设计

PR 3: 测试先行——为新功能写好测试用例（暂时标记为跳过）
      → 通过测试用例明确预期行为，审查者通过读测试就能理解需求

PR 4-N: 逐步实现功能，每个 PR 让一批测试通过
```


这是最容易被忽视但最有效的技巧。很多大 MR 中一半的差异来自文件移动和函数提取，这些噪音会淹没真正重要的逻辑变更。把它们单独提交，后续 PR 的差异就会干净很多。

## 第四部分：「但这个真的拆不了」


几乎每个大 MR 的作者都会说这句话。以下是常见的理由和对应的解法：

### 「必须一起改才能编译通过」


**解法**：引入中间过渡状态。

你的最终目标是从状态 A 到状态 C，但中间可以经过 A → B → C，其中 B 是一个「新旧代码并存」的过渡状态。旧接口暂时保留并标记为即将废弃，新接口先加进来，下一个 PR 再切换调用方并删除旧接口。

```TypeScript
// PR 1: 新增新接口，保留旧接口
/** @deprecated 请使用 newMethod 替代，将在下个迭代移除 */
function oldMethod() { /* ... */ }
function newMethod() { /* ... */ }

// PR 2: 迁移调用方到新接口
// PR 3: 移除旧接口
```

每个 PR 中代码都能正常编译和运行。

### 「必须看到全貌才能验证架构」


**解法**：用原型分支验证架构，用小 PR 正式合入。

做一个完整的原型分支（不需要达到上线标准），在分支上验证架构可行后，将原型分支作为参考蓝图，重新按小 PR 拆分并合入主干。

这两步的工作量看似增加了，但总时间通常更短——因为小 PR 合入快、审查快、冲突少。

VFS PR 的作者实际上也走了类似的路——他先做了 130 个 commit 的完整分支，后来转回 Draft 做重构。如果从一开始就计划好「先写原型验证可行性，再拆分合入」，流程会顺畅得多。

### 「拆成小 PR 后每个都不算完整功能」

**解法**：区分「对用户有价值」和「对工程有价值」。

一个类型定义的 PR、一个被 feature flag 隐藏的半成品功能，对终端用户没有直接价值。但对工程过程有巨大价值——它让代码尽早进入主干、尽早接受审查、尽早暴露问题。

只要每个 PR 合入后代码仍然可以正常编译和运行（即使新功能还不可用），它就是一个合格的 PR。

### 「数据库结构变更必须和代码一起提交」

**解法**：向前兼容的分步迁移。

```Plain
PR 1: 添加新字段（允许为空或设定默认值），代码暂不使用该字段
PR 2: 代码开始写入新字段，同时继续写入旧字段（双写）
PR 3: 运行迁移脚本，将历史数据填充到新字段
PR 4: 代码切换到读取新字段
PR 5: 移除旧字段和双写逻辑
```


每一步都是安全的、可以独立回滚的。这也是 GitHub、Shopify 等大型团队处理数据库迁移的标准做法（参见 GitHub 工程博客：[gh-ost: GitHub's online schema migration tool](https://github.blog/news-insights/company-news/gh-ost-github-s-online-migration-tool-for-mysql/)）。

## 第五部分：实操检查清单

提交 MR 前，对照以下清单自检：


- 改动行数（不含自动生成代码和测试）是否在 500 行以内？
    
- 能否用一句话描述这个 MR 做了什么？
    
- 审查者能否在 30 分钟内完成审查？
    
- 如果线上出问题，能否安全回滚这一个 MR 而不影响其他功能？
    
- MR 中是否同时包含了重构和新功能？（应拆分为两个 PR）
    
- MR 中是否包含了多个不相关的改动？（每个独立改动应单独提交）
    
- 是否有可以先独立合入的类型定义、接口声明或测试用例？
    
- 如果是大功能的一部分，是否有 feature flag 保护新代码不被用户触达？
    


如果以上任一项答案为「否」或「不确定」，优先考虑拆分。

## 第六部分：给不同角色的建议

### 给 MR 提交者


- **拆分是你的责任，不是审查者的。** 审查者不应该需要反过来请求你拆分
    
- 写好 MR 描述：说清楚「为什么做这个改动」而不仅是「改了什么」
    
- 如果是系列 PR 的一部分，在描述中说明关系：「这是 X 功能的第 3/7 个 PR，前置依赖是 #123」
    
- 站在审查者的角度思考审查路径：他应该先看哪个文件？改动的核心决策在哪里？
    

### 给审查者

- 遇到大 MR 时，可以礼貌地要求拆分：「这个 MR 范围较大，能否先把 X 部分拆出来？这样我可以更快给你反馈」
    
- 如果确实无法拆分，要求提交者提供审查指引：先看哪个文件、核心决策在哪里、哪些文件是自动生成可以跳过
    
- 不要因为 MR 大就降低审查标准——如果审查不过来，坦诚地说出来，比假装审查了更负责
    
### 给技术负责人

- 把 MR 大小作为团队代码审查文化的一部分来建设，而不仅是个人习惯
    
- 考虑在 CI 中设置提醒：超过一定行数时自动提示「建议考虑拆分」
    
- 关注「MR 从提交到合入的平均周期」这一指标——它与 MR 大小高度正相关（参见 [DORA 研究](https://dora.dev/research/)：交付周期是衡量团队效能的四个关键指标之一）
    
- 为复杂功能建立设计文档或 RFC 流程，确保在编码前达成设计共识

## 总结


**小 MR 不是额外的工作量，而是降低总工作量的手段。**

拆分 MR 的能力是工程成熟度的体现。初级工程师写出能运行的代码，中级工程师写出可维护的代码，高级工程师确保代码能以最低风险、最高效率地进入生产环境——而 MR 的组织方式，正是这最后一环中最容易被低估的技能。
